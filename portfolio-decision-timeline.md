# 포트폴리오 설득력 강화용 설계 의사결정/타임라인

작성일: 2026-04-20

이 문서는 “무엇을 만들었는가”보다 “왜 이렇게 설계했고, 어떤 문제를 줄였는가”를 설명하기 위한 포트폴리오용 문서다.

## 1. 문제 정의

처음부터 목표를 “기능 동작”이 아니라 아래 3가지로 잡았다.

1. 예산 정합성: 동시 요청과 실패 상황에서도 예산 상태를 일관되게 유지
2. 추적 신뢰성: bid/win/imp/click를 재구성 가능한 키로 연결
3. 구조 유지성: 빠른 구현 중에도 경계가 무너지지 않게 관리

## 2. 핵심 의사결정

## 2.1 식별자 분리: `requestId` vs `auctionId`

- 배경
  - 외부 `requestId`를 내부 mutation key로 그대로 쓰면 중복/재시도/오염에 취약
- 결정
  - 외부 연계용은 `requestId`
  - 내부 상태 변경/조인용은 `auctionId`
- 효과
  - 예산 예약 키의 안정성 향상
  - 퍼널 분석 기준 키 단순화

## 2.2 예산 처리: Redis Lua 예약/확정/환불

- 배경
  - DB 트랜잭션만으로 고속 비딩 경로의 예산 경쟁을 처리하기 어려움
- 결정
  - Redis 키 + Lua로 reservation/confirm/refund 분리
- 효과
  - 예산 상태 변경의 원자성 확보
  - 비딩 핫패스에서의 상태 판단 단순화

## 2.3 tracking 최소화

- 배경
  - auction 단위로 긴 문자열(`clickUrl`)을 저장하면 메모리 비용 증가
- 결정
  - `auction_tracking`에는 최소 필드만 저장
  - `clickUrl`은 metadata projection 조회
- 효과
  - 메모리/키 churn 비용 절감
  - 추적 필수 정보는 유지

## 2.4 후속 이벤트 입력 단순화

- 배경
  - win/imp/click에 다수 파라미터를 받으면 위변조/불일치 가능성 증가
- 결정
  - `aid`만 받고 tracking 복원 기반으로 처리
- 효과
  - 입력 신뢰 경계가 단순해짐
  - 컨트롤러 책임 축소

## 2.5 ad_manager의 복잡도 제어

- 배경
  - 지원 서비스가 과도한 이벤트/재시도 인프라를 가지면 설명비용이 커짐
- 결정
  - 생성(`inactive`)과 활성화 분리
  - DB -> Redis 동기 호출, 실패 시 보상 트랜잭션
- 효과
  - 핵심 실패 경로를 짧고 명확하게 설명 가능
  - `bidder` 중심 포트폴리오 메시지 유지

## 3. 작업 타임라인

## 3.1 MVP 단계

- WebFlux + R2DBC 기반 기본 비딩 흐름 구현
- 기능 완성을 먼저 확보

## 3.2 이벤트 아키텍처 단계

- Kafka Connect 실험 후 직접 consumer 방식으로 전환
- Avro + Schema Registry 적용
- producer(`bidder`) / sink(`log-consumer`) 분리

## 3.3 경계 정리 단계

- 헥사고날 경계 규칙 문서화
- 컨트롤러가 outbound에 직접 의존하지 않도록 정리

## 3.4 식별자/추적 재설계 단계

- `auctionId` 도입
- reservation 키 전환
- 후속 엔드포인트 `aid` 기반 전환
- tracking 복원 구조 반영

## 3.5 시간축 정리 단계

- `receivedAt`(요청 수신 시각), `createdAt`(DB 저장 시각) 분리
- 이벤트/DB 시각 의미를 분명하게 고정

## 3.6 성능 기준선 단계

- k6 스크립트와 baseline 결과 파일 추가
- 개선 전후 비교 가능한 최소 측정 자산 확보

## 4. 이 포트폴리오에서 강조하면 좋은 포인트

1. “외부 계약 신뢰”와 “내부 불변조건”을 분리해서 다룸
2. 고속 경로에서 정합성과 비용(메모리/churn)을 함께 고려함
3. 설계 문서가 코드 변경으로 이어진 흔적이 명확함
4. 실패를 숨기지 않고, 남은 과제를 명시적으로 관리함

## 5. 면접/리뷰에서 바로 말할 수 있는 문장

1. “`requestId`는 correlation이고, mutation key는 `auctionId`로 분리했습니다.”
2. “예산 상태 변경은 Redis Lua로 reservation/confirm/refund를 나눠 원자성을 확보했습니다.”
3. “tracking은 최소 정보만 저장하고, 무거운 메타데이터는 projection 조회로 분리해 비용을 낮췄습니다.”
4. “ad_manager는 운영 플랫폼으로 키우기보다 bidder 보조 서비스로 복잡도를 제어했습니다.”
5. “현재는 구조가 강점이고, 운영 마감도(log-consumer 보존/중복/관측성)가 다음 개선 포인트입니다.”

## 6. 현재 상태를 솔직하게 보여주는 보완 과제

1. `log-consumer`의 `ddl-auto`, idempotency, 관측성 개선
2. 일부 테스트 실행 안정성 정리
3. bidder 핫패스 최적화와 성능 결과 재측정

위 3가지를 보강하면, 이 프로젝트는 “설계가 좋은 학습 프로젝트”에서 “운영 감각까지 갖춘 실무형 포트폴리오”로 더 설득력 있게 확장된다.
